Available Datablock Types

The different Datablock Types optionally control special handlings for the respective datablocks.

Type Description
DRIVER1 The flash driver has to be downloaded since the erase and flash routines do not permanently reside in the ECU
DRIVER2 If DRIVER2 is available, DRIVER1 usually only contains the erase routine and DRIVER2 contains the flash routine
DATA All data files which shall be flashed

Downloading several drivers is supported. Optionally the drivers may even be different.

Logical Blocks

The logical block areas are erased and programmed using addresses.

ODX Own-Idents

A datablock may contain a list of Own-Idents that can be used to check the version of the data. E.g. Supplier or Software Version.
If Own-Idents are available for a Datablock and are equal with the Own-Idents reported by the ECU in the Identify Step the Datablock will skipped.

For this feature to work it is necessary that the ODX/PDX files define Expected Idents as well.
It may be necessary to disable this feature (e.g. in case of bootloader update or updates where application memory is used to write temporary dara). This can be achieved by activating the Option Flash also sessions identified as non-matching in the 'Select' Tab.

Checksum / Signature

  • Checksum or Signature: If either checksum or signature is defined for a datablock the routine Check Memory 0x31 01 02 02 is send with the defined data.
  • Checksum and Signature: If a checksum and signature file is defined for a datablock the routine Check Memory 0x31 01 02 02 is only send with the checksum data.
  • Embedded Checksum/Signature: If no checksum or signature is specified for the flashware, it is assumed checksum or signature is embedded within the flashware and the routine Check Memory 0x31 01 02 02 is send without additional data.

Additional Notes

  • In case of communication errors during TransferData requests are repeated once.

Communication Specifics

Logical Link

CAN / CAN FD

  • CAN ID Type usage: For functional and physical addressing different CAN ID Types (11 bit and 29 bit) may be used
  • Extended Addressing: Please note that extended addressing is only relevant for the layout of CAN frame content. Extended addressing can be used in combination with 11 bit or 29 bit CAN identifiers, it does not change the identifiers

DoIP / DoIP IPv6

  • In case the ECU refuses Routing Activation e.g. with the negative response code Authentication Missing 0x04 the vFlash Template retries to initiate Routing Activation

Force Boot Mode

  • If an invalid application is loaded into an ECU it is usually not possible to start the bootloader, since the Diagnostic Session requests are not processed anymore. With the Force Boot Mode feature the bootloader can be detained from loading the invalid application and stay in the bootloader.

Miscellaneous

Tester Serial Number

  • The Tester Serial Number must be not more than 100 ASCII characters long.
  • The Tester Serial Number is written to the ECU as part of the fingerprint write step. It will be transfered to the ECU in combination with the current date.
